## 核心观点

1. **记忆三分法**（Tulving 1972）：情境记忆、语义记忆、程序化记忆
2. **Claude Code Memory 实践**：三层记忆对应实现
3. **框架对比**：全量记忆 vs 执行导向

## 详细内容

### 记忆系统的三分法

Tulving 老爷子在 1972 年提出了记忆系统的框架设计，至今仍在广泛使用，他把记忆分为三类：

1. **情境记忆（episodic memory）** - 我干了啥，用来记录过去
2. **语义记忆（semantic memory）** - 我知道啥，用来提炼规律
3. **程序化记忆（procedural memory）** - 我会做啥，用来决定行动

这些记忆之间是存在相关性的，构成了一个不断循环的过程：**经历 → 知识 → 技能 → 新的经历**

### Claude Code Memory 的实践

**情境记忆**：每轮次的对话以 jsonl 格式做储存

**语义记忆**：每轮次结束后 fork 出一个子 Agent（extractMemories）去提取值得持久化的内容，写入带 YAML 头的 MD 文件，把零散对话压缩成可复用的知识

**程序化记忆**：feedback 的记忆类型，不仅记录了不要做什么（用户的纠正），还会记录哪些做法是对的（用户的确认）

**autoDream 机制**：在后台做整合和修剪，把碎片化的内容重新组织、合并、更新，很像记忆的巩固过程

### 与其他框架对比

**全量记忆框架**（LangMem/Mem0/Zep/EverMemOS/MemOS）：
- 追求全量记忆
- 工程和架构复杂度直线上升
- 基础设施包含 MongoDB/Elasticsearch/Milvus/Redis
- 更适合做个人长期记忆管理

**OpenClaw**：
- 不追求全量记忆
- 优先解决"记忆什么时候该被用"
- 三层记忆：全局层、工作区层、任务层
- 逐层收敛，只在必要的时候才把上下文拉进来
- 记忆不是资产，正确使用记忆的能力才是

**Claude Code**：
- 选了一条更偏执行的路径
- Memory 的作用是"防遗忘机制"
- 保证关键上下文不会在多轮试错中丢失
- 更像人类工程师 debug：一边看、一边改、一边跑、一边修正

---

## 我的思考

- Claude Code 的记忆设计虽然简单，但实用
- OpenClaw 的三层记忆设计更灵活
- 全量记忆框架复杂度高，适合长期记忆管理
- 执行导向的记忆更适合任务完成
